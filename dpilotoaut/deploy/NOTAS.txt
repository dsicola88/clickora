Deploy — Railway (app + Postgres) e Vercel (fullstack com Nitro)
================================================================

Requisitos gerais
-----------------
- Node >= 22.12 (package.json "engines")
- Variáveis no .env de produção (ver env.example): DATABASE_URL, AUTH_SECRET, CRON_SECRET, chaves OAuth e IA conforme use
- Build local: npm run build (Nitro gera frontend/.output para Node; na Vercel com VERCEL=1 gera frontend/.vercel/output)
- Migrações: backend/prisma/migrations/ (inicial: 20260425120000_init). Em produção: `npm run db:migrate:deploy` (já no Railway preDeploy e no build Vercel).

Railway — serviço Web + plugin PostgreSQL
------------------------------------------
1. Novo projeto → GitHub → este repositório.
2. Adicione o plugin "PostgreSQL"; copie a variável DATABASE_URL para o serviço web (ou use a referência automática).
3. Defina variáveis: AUTH_SECRET (min. 16 chars), CRON_SECRET, e as de integrações que precisar.
4. O ficheiro railway.toml define:
   - build: prisma generate + npm run build
   - preDeploy: prisma migrate deploy
   - start: node frontend/.output/server/index.mjs
   - healthcheck: GET /hooks/health
5. NIXPACKS_NODE_VERSION=22.12.0 (ou 22) nas variáveis do serviço se o build reclamar da versão do Node.
6. Base nova: o primeiro deploy aplica a migração inicial (preDeploy) sobre uma base vazia.
7. Bases antigas criadas só com `prisma db push` (dev local): o esquema pode já existir. Marcar migração como aplicada (se tiver a certeza de que bate com o ficheiro SQL):
   npx prisma migrate resolve --applied 20260425120000_init --schema backend/prisma/schema.prisma
8. Uploads Meta: UPLOAD_DIR é efémero; para persistir, monte um volume no Railway ou use armazenamento externo (S3, etc.).
9. OAuth (Google/Meta/TikTok): URLs de redirect no painel de cada plataforma devem ser https://<o-seu-dominio>/hooks/... (e GOOGLE_OAUTH_REDIRECT_URL / META_OAUTH_REDIRECT_URL / TIKTOK_OAUTH_REDIRECT_URL em produção).

Vercel
------
1. Importe o repositório; raiz do projeto = raiz do repo (onde está vercel.json).
2. O ficheiro vercel.json define build: prisma generate + migrate deploy + npm run build, output frontend/.vercel/output, framework tanstack-start.
   O build precisa de DATABASE_URL (Vercel injeta a variável em build time).
3. Na Vercel, o build define VERCEL=1 → o Nitro usa o preset "vercel".
4. Configure as mesmas variáveis de ambiente (DATABASE_URL pode ser a do Railway Postgres — URL pública com SSL, ou Vercel Postgres).
5. Domínio: defina VITE_APP_PUBLIC_URL se a app usar URLs absolutas no cliente.
6. Serverless + Prisma: considere connection_limit baixo na DATABASE_URL (ex. ?connection_limit=5&pool_timeout=10) ou Prisma Accelerate para muitas instâncias.

Comandos úteis
--------------
Ver deploy/TESTE-LOCAL-PROD.txt  —  como alinhar teste local com produção (mesmo migrate deploy + build).

npm run dev                # desenvolvimento (HMR)
npm run db:migrate         # cria e aplica migrações a partir de schema (dev, base local)
npm run db:migrate:deploy  # aplica só migrações pendentes (igual a produção)
npm run verify:predeploy  # generate + migrate deploy + build (replica pipeline antes de servir)
npm run test:local:prod   # igual a verify:predeploy
npm run test:vercel:build # gera saída estilo Vercel (VERCEL=1)
npm run build:production  # generate + build (sem migrate; use verify:predeploy para paridade)
npm run start              # após build: servidor Nitro (PORT definido por ti ou pelo hosting)
npm run preview            # pré-visualização Vite (alternativa local)
